# 决策记录（陪读小精灵）

按对话逐步收敛的范围与选型，记下"为什么"，避免以后反复。

## 1. 为什么用 ESP32（而非树莓派）
- 架构是"全走云端"：本地只当**采音/播音/联网转发**的瘦客户端 —— 正是 ESP32 的主场。
- 便宜、低功耗（可电池）、瞬间开机；契合作者的**裸机/嵌入式**背景。
- 代价：断网即死（本地跑不了任何模型）。用于"家里书桌 + 有 WiFi"场景可接受。

## 2. 为什么必须 ESP32-S3 + PSRAM（不用老 ESP32）
- HTTPS/TLS 握手吃内存（~40KB）；音频缓冲 + JSON 也要空间。
- 去掉摄像头后内存压力已小很多，但 S3+PSRAM 仍更稳。

## 3. 范围收敛过程
- 台灯 → 砍掉外形，只做"陪读/聊天"。
- 拍照批改 → **砍掉摄像头**：手写字批改对**画质/对焦/光照**要求高，OV2640 2MP 定焦大概率糊，是最大风险。去掉它，项目难度骤降。
- 坐姿检测 → 一并砍掉（依赖摄像头）。
- 结论：**纯语音陪伴 + 极简水墨屏**。

## 4. 屏幕选型：水墨屏 vs 圆形 IPS
- 水墨屏（选定）：护眼、省电、纸感；**擅长静态文字 + 偶尔换表情**，不擅长动画。
  - 表情 = 几张预画位图（idle/听/想/说），状态切换时**局部刷新一次**即可，别做连续动画。
- 圆形 IPS（备选）：GC9A01 240×240，能做丝滑"大眼睛"桌宠动画，但有背光、耗电、无护眼卖点。
- 决策：主打护眼安静 → 用水墨屏；把"生动"交给**声音（TTS 语气）**而非屏幕。

## 5. 语音链路关键技术选择
- **TTS 请求 PCM/WAV，不要 MP3**：ESP32 免解码，直接喂 I2S 功放播放。
- **先 push-to-talk（按住说话）**：录完再传（火山 submit/query，~2s 出字）；成熟后再上流式 ASR(wss)。
- 模型分工（复用 ai_lamp 经验）：陪聊用 pro（seed-2-1-pro），需要快/省时用 flash（seed-1-6-flash）。
- 密钥走 .env（gitignore），沿用 ai_lamp 的零依赖 env 加载约定。

## 6. 开发路线
1. 先在 Pi/PC 上用 Python 把"按住说话 → ASR → 豆包 → TTS(PCM) → 播放"闭环调定型（提示词/节奏）。
2. 定稿后移植到 ESP32-S3：原型可 MicroPython（作者 Python 熟、避开 socket/RTOS 细节），产品转 ESP-IDF/C。

## 7. 与 ai_lamp 的关系
- 姊妹项目。云端语音/豆包的踩坑经验（TTS 走 V3+volc.tts.default、ASR 走 volc.seedasr.auc 等）直接复用。
- ai_lamp 保留"台灯 + 摄像头 + 坐姿"完整设想；本项目是其"纯语音"精简分支。

## 8. 网络架构：自建 server + 中转站（含零信任约束）
### 为什么要"先自己 server 再走云"
- 需要**记录对话、二次处理、做总结** → 云端 API 只做单次调用、不存历史，所以链路里**必须**有一台自己的后端做编排与持久化。
- 因此 ESP32 永远只连"自己的后端"，不直连豆包（也顺带保护密钥，别把 API Key 烧进固件——ai_lamp 已被欠费坑过）。

### 零信任约束（关键事实）
- 最终生产 server 在**另一家公司**，与家里**不同网段**，访问受 **aTrust（深信服 ZTNA）** 管控。
- **ESP32 无法接入 aTrust**：aTrust 是"给人用"的客户端式 ZTNA（客户端 + SPA + SSO 登录），协议闭源、无嵌入式 SDK。设备装不了、也登不了。
- aTrust 客户端架构：**Android 有**、**x86 Linux 有**、**ARM Linux 基本没有** → 选中转硬件时据此排除 ARM SBC（树莓派/香橙派跑不了 aTrust）。

### 平面分离（设备 ≠ 人）
- **人的平面**：作者用 aTrust 客户端（电脑/手机）登进公司 server 看总结、做管理。
- **设备平面**：ESP32 用机器身份（mTLS/token）连**中转站**，绝不借人的 aTrust 身份穿隧道。
- 合规红线：用个人 aTrust 账号替设备转发数据 = "拿人的身份给设备当代理"，产品化阶段禁止；**仅自用可临时接受**。

### 中转站方案（两阶段，硬件可换、上层代码不变）
- **前期（自用）**：一台**闲置二手安卓手机**当中转站——装 aTrust 安卓客户端 + 一个常驻转发程序，息屏插电、专机专用。成本 ¥0，ARM 无所谓（用安卓客户端非 Linux 客户端）。
- **后期**：换一台 **x86 mini 主机**（或二手瘦客户端 HP t620/Dell Wyse，¥100 出头）常开当中转，x86 保证 aTrust Linux 客户端可装，7×24 稳定、闲时 5–10W。
- **产品化（面向别人家）**：回归标准做法——设备走**自己的公网 broker（推荐 MQTT over TLS）+ 机器身份**，两边都主动外连，不靠任何人的 aTrust。

### 开发期做法（现在）
- 直接用 **4090** 当中转/后端（作者当前能连公司网），先把 `host/` 后端闭环跑通。
- **后端地址做成配置项**：从"4090 直连"→"安卓手机中转"→"mini 主机"→"公网 broker"全程只改配置，架构不动。
